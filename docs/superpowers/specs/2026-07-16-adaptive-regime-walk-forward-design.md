# Дизайн адаптивной режимной walk-forward стратегии

Дата: 2026-07-16  
Статус: одобрено пользователем

## 1. Цель

Создать отдельный исследовательский walk-forward бэктест, который ежедневно выбирает один из трёх режимов работы существующего новостного сигнала:

- **импульс** — торговать в направлении исходного сигнала;
- **контртренд** — торговать против исходного сигнала, как в текущем `simulate_trade.py`;
- **вне рынка** — не открывать позицию при недостаточной уверенности.

Гипотеза исследования: ухудшение RTS после 2026-05-03 связано не с исчезновением сходства новостей, а со сменой реакции рынка с контртрендовой на импульсную. Новый бэктест должен проверить эту гипотезу без заглядывания в будущее для RTS, MIX, NG, BR, GOLD, SI и SPYF.

## 2. Границы задачи

Исследовательский контур должен быть полностью изолирован. Он не изменяет:

- семь существующих `simulate_trade.py`;
- `run_enb_sim_an.py` и `run_all.py`;
- файлы прогнозов;
- скрипты живой торговли;
- существующие `df_rez_output.xlsx` и `strategy_analysis.html`.

Допустимые изменения:

- новый запускной скрипт и новые исследовательские модули;
- новый блок настроек `regime_walk_forward` в семи `settings.yaml`;
- новые тесты;
- отдельная документация и отдельные результаты исследования.

Комиссия и проскальзывание не учитываются. Результат измеряется в тех же пунктах, что и текущий бэктест.

## 3. Архитектура

Планируемые компоненты:

1. `run_adaptive_regime_walk_forward.py` — единая точка запуска. Поддерживает один тикер через `--ticker rts` и все тикеры через `--all`.
2. `research/adaptive_regime_walk_forward.py` — загрузка и валидация данных, расчёт режимного индикатора, подбор параметров и ежедневный walk-forward.
3. `research/adaptive_regime_report.py` — расчёт итоговых метрик, создание Excel и интерактивного HTML-отчёта.
4. `docs/adaptive_momentum_countertrend_walk_forward_strategy.md` — подробное пользовательское описание стратегии, формул, настроек, запуска, интерпретации и ограничений.
5. `tests/test_adaptive_regime_walk_forward.py` — модульные и интеграционные проверки.

Общая реализация используется всеми тикерами. Копии логики по папкам тикеров не создаются, чтобы расчёты и исправления оставались одинаковыми.

## 4. Входные данные

Для каждого тикера читается `{ticker}/df_rez_output.xlsx`. Обязательные поля:

- `TRADEDATE` — дата результата;
- `P/L` — P/L текущей контртрендовой стратегии после инверсии;
- `max` — выбранное текущим симулятором значение `k`, сохраняемое в отчёте для диагностики.

Текущий выбор `k` считается допустимым базовым сигналом, потому что `simulate_trade.py` рассчитывает `PL_k` со сдвигом на один день и выбирает `k` только по уже известным результатам.

Для дня `d` вводятся:

- `C_d = P/L_d` — результат контртрендового режима;
- `M_d = -C_d` — результат импульсного режима;
- `D_d = M_d - C_d` — преимущество импульсного режима над контртрендовым.

## 5. Настройки

В каждый тикерный `settings.yaml` добавляется независимый блок:

```yaml
regime_walk_forward:
  train_window_days: 63
  score_spans: [5, 8, 13, 21]
  confidence_thresholds: [0.05, 0.10, 0.20, 0.30]
  min_regime_days: [1, 3, 5]
  drawdown_penalty: 1.0
```

Назначение параметров:

- `train_window_days` — число последних торговых дней в обучающем окне;
- `score_spans` — кандидаты периода экспоненциального сглаживания;
- `confidence_thresholds` — кандидаты симметричного порога уверенности;
- `min_regime_days` — кандидаты минимальной длительности активного режима;
- `drawdown_penalty` — множитель штрафа за максимальную просадку в целевой функции.

Все параметры можно задавать отдельно для каждого тикера. Сетки должны быть непустыми, значения — конечными и положительными, пороги — не больше `1`, а период сглаживания — меньше обучающего окна.

## 6. Режимный индикатор

Для кандидата с периодом сглаживания `s` перед решением дня `d` рассчитываются только предыдущие наблюдения:

```text
mean_d  = EWM_s(D до дня d)
scale_d = EWM_s(abs(D) до дня d)
score_d = mean_d / max(scale_d, epsilon)
```

Нормирование делает индикатор безразмерным и сопоставимым между тикерами. Для расчёта требуется не менее `s` прошлых наблюдений; до окончания прогрева режим нейтральный.

Для порога `h` предварительное решение имеет вид:

- `score_d > h` — импульс;
- `score_d < -h` — контртренд;
- `-h <= score_d <= h` — вне рынка.

После входа в активный режим смена или выход в нейтральный режим запрещены в течение выбранного `min_regime_days`. Зафиксированный при входе срок не изменяется последующей переоптимизацией.

## 7. Вложенный walk-forward

Для каждого out-of-sample дня `t` выполняются следующие действия:

1. Формируется обучающее окно из `train_window_days` строк, заканчивающееся днём `t-1`.
2. Строится декартово произведение `score_spans`, `confidence_thresholds` и `min_regime_days`.
3. Каждый кандидат причинно симулируется внутри обучающего окна. Решение для любого учебного дня использует только более ранние строки этого окна.
4. В режиме импульса кандидату начисляется `M_d`, в контртренде — `C_d`, вне рынка — `0`.
5. Рассчитываются накопленный P/L, максимальная просадка и число переключений.
6. Кандидат оценивается по формуле:

   `objective = total_pl - drawdown_penalty * max_drawdown`

7. Выбирается кандидат с максимальной оценкой. При равенстве предпочтение получает меньшая просадка, затем меньшее число переключений; оставшееся равенство разрешается стабильным порядком параметров.
8. Полностью нейтральная стратегия с оценкой `0` участвует как защитный эталон. Если лучший активный кандидат не превосходит её, предварительное решение для дня `t` — вне рынка.
9. Выбранные параметры и индикатор, рассчитанный только по обучающему окну, определяют режим дня `t`.
10. Если ранее начатый активный режим ещё заблокирован минимальным сроком, он сохраняется. Иначе применяется новое решение.
11. После фиксации решения раскрывается фактический P/L дня `t`, записывается out-of-sample результат, окно сдвигается на один день, и подбор повторяется.

Первые `train_window_days` строк являются прогревом и не входят в итоговый out-of-sample результат. Таким образом, ни итог дня `t`, ни более поздние данные не влияют на решение дня `t`.

## 8. Результаты

Для каждого тикера создаётся каталог:

`{ticker}/research/adaptive_regime_walk_forward/`

В нём сохраняются:

- `results.xlsx`;
- `report.html`.

Книга `results.xlsx` содержит:

### Лист `daily`

- дата;
- исходный `max`;
- контртрендовый и импульсный P/L;
- предварительный и фактически применённый режим;
- режимный индикатор;
- выбранные `score_span`, `confidence_threshold` и `min_regime_days`;
- обучающие `objective`, P/L, просадка и число переключений;
- out-of-sample P/L;
- накопленный P/L и текущая просадка.

### Лист `summary`

- накопленный P/L;
- максимальная просадка;
- Profit Factor;
- win rate среди активных дней;
- число активных дней;
- доли импульсного, контртрендового и нейтрального режимов;
- число переключений;
- аналогичные показатели постоянного контртренда и постоянного импульса на том же out-of-sample периоде.

### Лист `config`

Снимок всех фактически использованных настроек и путей к входным данным.

HTML-отчёт содержит:

- накопленный P/L адаптивной стратегии, постоянного контртренда и постоянного импульса;
- дневной P/L;
- просадку;
- временную шкалу режимов;
- динамику подобранных параметров;
- итоговую таблицу метрик.

Общая книга `research/adaptive_regime_walk_forward_summary.xlsx` сопоставляет результаты всех тикеров и содержит статус `success`, `skipped` или `error` с пояснением.

## 9. Обработка ошибок

До расчёта проверяются существование файлов, обязательные столбцы, корректность и уникальность дат, конечность P/L, настройки и достаточность истории. Некорректные данные не исправляются скрытно: тикер получает понятный статус и причину.

При `--all` ошибка одного тикера не прерывает обработку остальных. Код возврата:

- `0`, если успешно рассчитан хотя бы один тикер;
- `1`, если не рассчитан ни один тикер или некорректны аргументы запуска.

Ollama, сеть и кэши эмбеддингов новому исследованию не требуются.

## 10. Проверки

Автоматические тесты должны покрывать:

- противоположность P/L импульса и контртренда;
- положительный, отрицательный и нейтральный режимы;
- прогрев индикатора;
- минимальную длительность режима;
- максимальную просадку и целевую функцию;
- детерминированное разрешение равенств;
- нейтральный эталон с оценкой `0`;
- отсутствие утечки: изменение P/L дня `t` не меняет решение для `t` и предыдущих дней;
- искусственную историю со сменой работающего режима;
- короткую историю, невалидные настройки и частичный сбой запуска `--all`;
- запись только в исследовательские пути.

После тестов выполняется реальный запуск для семи тикеров. Дополнительно проверяется:

- число out-of-sample строк и диапазон дат;
- равенство итогового P/L сумме дневных результатов;
- согласованность просадки с кривой капитала;
- создание Excel и HTML для успешных тикеров;
- отсутствие изменений в текущих симуляторах, прогнозах и торговых файлах.

## 11. Критерии завершения

Работа завершена, если:

1. Новый скрипт запускает один или все семь тикеров.
2. Все параметры режима подбираются только внутри текущего обучающего окна.
3. Размер окна изменяется через соответствующий `settings.yaml`.
4. Исследовательские результаты не перезаписывают текущий пайплайн.
5. Созданы подробный MD-документ, Excel-результаты и HTML-отчёты.
6. Автоматические тесты проходят.
7. Реальный запуск формирует сводку по всем доступным тикерам.

Положительный бэктест сам по себе не является разрешением на перенос стратегии в прогнозы или живую торговлю. Такое решение требует отдельной задачи и отдельного одобрения.
