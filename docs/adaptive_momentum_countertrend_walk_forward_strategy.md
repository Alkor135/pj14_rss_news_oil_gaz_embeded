# Адаптивная импульсно-контртрендовая walk-forward стратегия

## Статус стратегии

Стратегия является отдельным исследовательским бэктестом. Она не подключена к текущим `simulate_trade.py`, файлам прогнозов или живой торговле. Даже положительный результат на всей доступной истории не означает автоматического разрешения на торговое применение.

## Зачем нужна смена режима

Текущая стратегия выбирает окно похожих новостных дней причинным способом, после чего зеркально инвертирует результат исходного сигнала. На RTS этот контртрендовый подход работал до начала мая 2026 года: накопленная прибыль достигла примерно 39 350 пунктов 2026-05-03. После вершины последовало снижение примерно на 41 070 пунктов.

До вершины доля прибыльных активных дней составляла около 65,2%, а Profit Factor — около 1,74. После вершины показатели снизились примерно до 37,9% и 0,34. При этом медианная косинусная схожесть новостей изменилась незначительно. Это поддерживает гипотезу, что перестала работать не сама похожесть новостей, а постоянная контртрендовая трактовка сигнала: рынок перешёл к более импульсной реакции.

Новая стратегия проверяет эту гипотезу и каждый день выбирает одну из трёх трактовок уже рассчитанного сигнала.

## Три режима

### Контртренд

Используется P/L, который уже записан в `{ticker}/df_rez_output.xlsx`. Это результат текущей стратегии после зеркальной инверсии исходного сигнала.

### Импульс

Используется противоположный знак контртрендового P/L. Этот режим моделирует торговлю в направлении исходного сигнала.

### Вне рынка

P/L равен нулю. Режим используется, когда обучающее окно не подтверждает преимущество активной стратегии или нормированный индикатор находится внутри зоны неопределённости.

## Обозначения

Для торгового дня `d`:

```text
C_d = P/L контртрендовой версии
M_d = -C_d
D_d = M_d - C_d
```

`D_d` показывает, насколько импульсная трактовка была лучше контртрендовой. Положительное значение поддерживает импульс, отрицательное — контртренд.

## Причинный индикатор режима

Для кандидата с периодом экспоненциального сглаживания `s` перед решением дня `d` рассчитываются:

```text
mean_d  = EWM_s(D до дня d)
scale_d = EWM_s(abs(D) до дня d)
score_d = mean_d / max(scale_d, epsilon)
```

Внутри обучающей симуляции ряд `D` сдвигается на один день. Поэтому фактический результат дня `d` не входит в решение этого дня.

Нормирование на сглаженный абсолютный P/L делает индикатор безразмерным. Это позволяет использовать одну сетку порогов для инструментов с разным масштабом пунктов.

Для симметричного порога `h`:

```text
score_d > h       -> momentum
score_d < -h      -> countertrend
-h <= score_d <= h -> flat
```

Строгие неравенства означают, что значение точно на границе порога считается нейтральным.

## Минимальная длительность режима

После входа в импульс или контртренд стратегия удерживает этот режим не меньше `min_regime_days`. Пока срок не завершён, новый сигнал режима и нейтральная зона не могут преждевременно закрыть его.

Минимальный срок фиксируется в момент входа. Если на следующем walk-forward шаге будет выбран другой набор параметров, уже начатый срок не сокращается. Это защищает результат от ежедневного дрожания режима и от обхода ограничения повторной оптимизацией.

## Обучающее окно

По умолчанию используется 63 последних торговых дня. Значение хранится отдельно в каждом тикерном `settings.yaml`:

```yaml
regime_walk_forward:
  train_window_days: 63
  score_spans: [5, 8, 13, 21]
  confidence_thresholds: [0.05, 0.10, 0.20, 0.30]
  min_regime_days: [1, 3, 5]
  drawdown_penalty: 1.0
```

Размер окна можно изменить независимо для каждого тикера. Все значения `score_spans` должны быть меньше `train_window_days`.

## Вложенный ежедневный walk-forward

Для каждого тестового дня `t` выполняется полный новый цикл:

1. Берутся последние `train_window_days` строк, заканчивающиеся днём `t-1`.
2. Строятся все комбинации `score_spans`, `confidence_thresholds` и `min_regime_days`.
3. Каждый набор параметров причинно тестируется внутри обучающего окна.
4. Решение каждого учебного дня использует только более ранние дни того же окна.
5. Для кандидата строится дневной P/L с учётом трёх режимов и минимального срока удержания.
6. Рассчитываются суммарный P/L, максимальная просадка и число переключений.
7. Выбирается лучший кандидат по риск-скорректированной целевой функции.
8. Выбранный набор определяет режим только одного следующего дня `t`.
9. После фиксации решения раскрывается фактический P/L дня `t`.
10. Окно сдвигается на день, и все параметры подбираются заново.

Первые `train_window_days` строк являются прогревом. Они используются для первого обучения, но не входят в итоговый out-of-sample P/L.

## Целевая функция

Для каждого кандидата:

```text
objective = total_pl - drawdown_penalty * max_drawdown
```

При равенстве целевой оценки выбирается кандидат с меньшей максимальной просадкой, затем с меньшим числом переключений. Полное оставшееся равенство разрешается стабильным порядком параметров, поэтому повторный запуск на одинаковых данных детерминирован.

Полностью нейтральная стратегия участвует как защитный эталон с оценкой `0`. Если ни один активный кандидат не получил оценку выше нуля, предварительный режим следующего дня — `flat`.

## Почему в тесте нет утечки будущего

Защита обеспечивается на двух уровнях:

- внутри обучающего окна индикатор дня использует сдвинутые результаты и не видит текущий день;
- при выборе режима out-of-sample дня `t` обучение заканчивается строго на `t-1`.

Изменение P/L тестируемого дня может изменить только начисленный результат этого дня и последующие решения. Оно не может изменить режим, уже выбранный для этого дня, или более ранние решения. Это свойство закреплено автоматическим тестом.

## Источник данных

Скрипт читает готовый `{ticker}/df_rez_output.xlsx` с полями:

- `TRADEDATE` — дата;
- `P/L` — P/L текущей контртрендовой стратегии;
- `max` — выбранное текущим симулятором значение `k` для диагностики.

Существующий выбор `k` использует сдвинутую историю `PL_k`, поэтому принимается как причинный базовый сигнал. Новый скрипт не пересчитывает эмбеддинги, не обращается к Ollama и не создаёт прогнозы.

## Запуск

Один тикер:

```powershell
.venv\Scripts\python.exe run_adaptive_regime_walk_forward.py --ticker rts
```

Все семь тикеров:

```powershell
.venv\Scripts\python.exe run_adaptive_regime_walk_forward.py --all
```

Поддерживаются `rts`, `mix`, `ng`, `br`, `gold`, `si` и `spyf`. Регистр аргумента не имеет значения.

Если один тикер не может быть обработан, общий запуск продолжает остальные. Код возврата равен `0`, если получен хотя бы один успешный результат, и `1`, если не рассчитан ни один тикер.

## Результаты одного тикера

Файлы сохраняются отдельно от текущего пайплайна:

```text
{ticker}/research/adaptive_regime_walk_forward/
├── results.xlsx
└── report.html
```

### Лист `daily`

Содержит дату, исходный `max`, P/L обоих постоянных режимов, предварительный и применённый режим, нормированный score, выбранные параметры, обучающие метрики кандидата, out-of-sample P/L, накопленную прибыль и просадку.

### Лист `summary`

Сопоставляет:

- `adaptive` — новый ежедневный режимный фильтр;
- `countertrend` — постоянную текущую инверсию;
- `momentum` — постоянную прямую трактовку.

Для всех вариантов используется один и тот же out-of-sample диапазон.

### Лист `config`

Хранит входной путь и снимок фактически использованных настроек. Это позволяет связать отчёт с конкретным запуском.

### HTML-отчёт

Содержит:

- три кривые накопленного P/L;
- дневной адаптивный P/L;
- просадку;
- временную шкалу режимов;
- динамику выбранных параметров;
- сравнительную таблицу итоговых метрик.

Общая сводка всех тикеров сохраняется в:

```text
research/adaptive_regime_walk_forward_summary.xlsx
```

## Метрики

- `total_pl` — сумма дневного P/L в пунктах;
- `max_drawdown` — наибольшее снижение от предыдущего пика кривой;
- `profit_factor` — сумма положительных активных дней, делённая на абсолютную сумму отрицательных;
- `win_rate` — доля прибыльных дней среди активных;
- `active_days` — число дней с импульсным или контртрендовым режимом;
- `mode_share_*` — доля времени в каждом режиме;
- `switches` — число фактических изменений режима.

Главное сравнение — не абсолютная прибыль сама по себе, а преимущество `adaptive` над обеими постоянными трактовками при сопоставимой или меньшей просадке.

## Комиссия и проскальзывание

Комиссия и проскальзывание намеренно не учитываются. Это сохраняет сопоставимость с текущим `df_rez_output.xlsx`, но делает результат исследовательским и потенциально завышает реальную доходность при частых переключениях.

## Ограничения

1. Доступная история коротка и содержит ограниченное число смен режима.
2. Параметры выбираются из заданной сетки; другая сетка может дать другой результат.
3. Нейтральный режим снижает число сделок и может пропускать быстрые развороты.
4. Входной сигнал наследует все ограничения текущего поиска похожих новостей и выбора `k`.
5. Не учитываются торговые издержки, ликвидность и ограничения исполнения.
6. Результаты разных фьючерсов выражены в пунктах и не являются напрямую сопоставимой денежной доходностью.
7. Один успешный исторический период недостаточен для вывода о стабильности стратегии.

## Условия перехода к следующему этапу

Стратегию можно отдельно рассматривать для более строгого исследования, только если:

- адаптивный вариант улучшает результат не на одном, а на нескольких тикерах;
- преимущество сохраняется при разумном изменении окна и сеток параметров;
- максимальная просадка не ухудшается относительно постоянных режимов;
- результат не формируется несколькими единичными днями;
- смены режима выглядят экономически объяснимыми;
- дополнительная история подтверждает поведение вне текущего периода.

Подключение к прогнозам и живой торговле потребует отдельной задачи, проверки издержек и явного одобрения.
